## 4. Why Do You Prefer Playwright For Enterprise Automation?

Playwright is strong for enterprise automation because it supports cross-browser testing, parallel execution, auto-waiting, tracing, screenshots, videos, network interception, API testing, storage state, and reliable CI execution.

Compared to older UI automation approaches, Playwright reduces flakiness when used properly with good locators, no hard waits, isolated test data, and meaningful assertions. With TypeScript, it also becomes easier to maintain a large framework.

## 5. How Do You Decide Whether To Test Through UI, API, Or Database?

I choose the test level based on risk, speed, and reliability. If the user journey or visual behavior is important, I test through UI. If the business logic can be validated faster through backend responses, I prefer API testing. If the feature is data-heavy, I validate database or source data as well.

For example, for a dashboard value, I may validate the UI value, API response, and database/source calculation. This catches issues where the UI renders correctly but the underlying data or aggregation is wrong.

## 6. How Do You Test Dashboard Or Analytics Features?

For dashboards, I validate filters, date ranges, counts, sorting, empty states, partial data, aggregation logic, and role-based visibility. I compare UI values with API responses or database/source values.

If analytics events are involved, I validate event emission, payload correctness, ingestion, aggregation, and final dashboard display. This is important because analytics systems can fail silently due to missing events, duplicate events, incorrect aggregation, or delayed ingestion.

I would describe Ella as an AI assistant with RAG and agentic/tool-calling capabilities. A simple RAG chatbot mainly retrieves context and generates an answer. Ella-like workflows can go further by using MCP servers or tools to fetch structured data, perform actions, or support user workflows.
